Amazon Rekognition este un serviciu de analiza si procesare de imagini ce foloseste deep-learning. Acest serviciu poate fi folosit cu usurinta drept "secondary authentication system", pentru a imbunatati accesul la aplicatie din punct de vedere al securitatii. Modul de identificare a unui utilizator, daca acesta este sau nu proprietarul de cont, va fi deci foarte simplu, si poate fi descris prin urmatorii pasi:

  • In momentul autentificarii se face o poza utilizatorului care doreste sa se conecteze, folosind camera WEB
  • Poza va fi comparata cu pozele asociate respectivului utilizator, iar drept raspuns va fi primit un obiect de tip JSON cu diferite metrici de interes precum "Similaritaty", "Confidence", acestea doua fiind suficiente pentru a confirma identitatea utilizatorului
  • In cazul unei recunoasteri cu succes a utilizatorului, noua poza pe care a fost testata identitatea va fi incarcata in folderul utilizatorului ca algoritmul sa invete in continuare (sa-si creasca gradul de precizie, confidenta)

Foloseste "Cryptography pseudo number generator", deci aduce un plus de securitate

Pune la dispozitie metode generice pentru criptare / decriptare a datelor etc.

Pune la dispozitie diferiti algoritmi de criptare: AES, RSA, SHA etc.

Securitatea Datelor

Formularul utilizatorului va fi criptat la nivel de client, inainte sa fie trimis in internet, cu ajutorul algoritmului de criptare AES. Cheia AES, generata random, si vectorul de initializare, de asemenea generat random, vor fi criptate si ele cu ajutorul cheii RSA publice a detinatorului site-ului. Formularul impreuna cu cheia AES si vectorul de initializare vor calatori prin internet, criptate, catre server prin intermediul protocolului HTTPS, acesta oferind un nivel de securitate suplimentar. Odata ajunse la server, formularul, cheia AES si vectorul de initializare vor fi trimise din nou prin internet, criptate, prin intermediul protocolului HTTPS catre S3 unde vor fi si stocate. S3 furnizeaza un nivel ridicat de incredere din punctul de vedere a confidentialitatii si a integritatii datelor.



La procesul de recuperare a formularelor de catre detinatorul site-ului, acestea sunt extrase din S3 impreuna cu cheia AES si vectorul de initializare, criptate, si trimise catre client, prin internet prin intermediul HTTPS. Decriptarea se face la nivel de client. Cheia RSA calatoreste de la server spre client prin intermediul HTTPS in cazul in care detinatorul site-ului alege optiunea in care cheia privata sa fie salvata si administrata de serviciul nostru. In caz contrat, cheia privata RSA nu paraseste clientul.

Posibile vulnerabilitati de securitate

In momentul in care serviciul nostru primeste un formular, trebuie sa identificam url-ul de pe care a fost completat acel formular, utilizatorul pentru care trebuie salvat si cheia cu care a fost criptat. Daca un atacator ar putea trimite o cerere care la server ajunge ca si cum ar fi trimisa de pe acel url whitelisted de catre detinatorul site-ului, atunci atacatorul ar putea cauza multe probleme, una dintre ele fiind umplerea bazei de date cu formulare redundante.

Single Page Application

Am ales o arhitectura de tip Single Page Application (SPA) pentru a oferi utilizatorilor o experienta mai placuta, mediul fiind mai asemanator astfel cu cel oferit de o aplicatie desktop. Astfel, tot codul necesar (HTML, JavaScript, CSS) va fi downloadat la prima interactiune a utilizatorului pe site si ulterior, resursele noi / necesare vor fi incarcate dynamic, fiind corespunzatoare actiunilor utilizatorului.

La nivel de server vom aborda metoda "Thin server architecture", iar logica de pe server va fi mutata la client, pe cat posibil, serverul ramanand astfel un API de trimitere si primire a datelor. Vom incerca sa imbinam metoda anterior mentionata cu un altfel de tip de arhitectura, "Thick stateless server architecture", iar la fiecare noua accesare a site-ului datele vor fi incarcate ca si cum ar fi prima oara (prima accesare), nerecurgand la cache-uirea unor parti de logica / date intrucat in primul rand este vorba despre date cu continut sensibil, si in al doilea rand, nu este necesara o putere computationala substantial crescuta. Astfel o noua utilizare va fi de fapt o noua stare a utilizatorului in aplicatie.

In concluzie, in cadrul interactiunii utilizatorilor cu pagina se va pune accent pe comunicare dinamica intre client si server, fiind rescris continutul paginii si nu incarcata o noua pagina.

Sunt astfel evitate intreruperile experientei utilizatorilor in succesiunea paginilor.

Doar o incarcare (mai consistenta) a codului la prima accesare a paginii, insa o experienta mai rapida in restul activitatii din pagina.

Intrucat logica va fi mutata (pe cat posibil) de la server la client, acest lucru, per total, va reduce complexitatea serverului.
